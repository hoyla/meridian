# 2026-05-19 — Ollama `/api/chat` 404 storm on `--periodic-run`

## Symptom

`python scrape.py --analyse llm-framing` (or full `--periodic-run`) failed
4×404 across the retry-with-backoff:

```
INFO httpx: HTTP Request: POST http://localhost:11434/api/chat "HTTP/1.1 404 Not Found"
WARNING llm_framing: Ollama transient error (attempt 1/4, status 404): model 'qwen3.6:latest' not found ...
... (3 more) ...
ERROR llm_framing: LLM error for group 'EV batteries (Li-ion)'
ollama._types.ResponseError: model 'qwen3.6:latest' not found (status code: 404)
```

`ollama list` showed `qwen3.6:latest` present. `ollama ps` was empty
before AND after — the model never loaded in response to the failing
requests.

The same call shape from a fresh `python -c "import llm_framing;
llm_framing.detect_llm_framings(...)"` (or `import ollama; ollama.chat(...)`)
succeeded reliably. The model loaded, `ollama ps` showed it on GPU.

## Root cause

Two listeners on port 11434:

```
COMMAND     PID         USER   FD   TYPE   NAME
com.docke 31377 luke_hoyland  177u  IPv6   *:11434          (LISTEN)
ollama    70888 luke_hoyland    3u  IPv4   127.0.0.1:11434  (LISTEN)
```

Docker Desktop's port forwarder was bound to `*:11434` over IPv6. The
host-native Ollama daemon was bound IPv4-only on `127.0.0.1:11434`.

`.env` had `OLLAMA_HOST=http://localhost:11434`. `localhost` resolves to
both `::1` and `127.0.0.1` on macOS; httpx (used inside the `ollama`
Python lib) tries `::1` first → request lands at Docker, not Ollama.
Whatever Docker was forwarding to returned `{"error":"model 'qwen3.6:latest'
not found"}` — exactly the response shape the Ollama daemon would emit
for a missing model, which is why the surface error looked like an
Ollama eviction race rather than a routing problem.

Verified by direct curl:

```
curl -X POST "http://[::1]:11434/api/chat" -d '{"model":"qwen3.6:latest", ...}'
→ HTTP 404 {"error":"model 'qwen3.6:latest' not found"}

curl -X POST "http://127.0.0.1:11434/api/chat" -d '{...}'
→ HTTP 200 (response from real daemon)
```

The direct `python -c` call succeeded because — with no `OLLAMA_HOST` in
the inherited env — the `ollama` Python client defaults to
`http://127.0.0.1:11434` (IPv4 literal, no DNS).

Why it broke today specifically: Docker's IPv6 listener appeared/started
binding port 11434 today. Prior cycles routed via IPv4 by default because
the IPv6 conflict didn't exist.

## Fix

Single-character change in `.env`: `localhost` → `127.0.0.1`.

`.env.example` updated the same way, with a comment explaining the
gotcha so a fresh checkout doesn't reintroduce the bug.

The retry logic in `llm_framing.OllamaBackend` was added under the
mistaken eviction-race hypothesis. Kept it — still useful for genuine
transients (model load 503/504, post-eviction reload windows) — but
updated the comment so the next reader doesn't inherit the wrong mental
model.

## Why the earlier "eviction race" hypothesis felt plausible but was wrong

- The response shape matched what Ollama emits for a missing model.
- The 4×404 retry didn't help, which looked consistent with a daemon
  that needed >22s to recover but inconsistent with eviction (load
  takes ~8s on this hardware).
- The "5 serial 404s in 25ms" anecdote in the original retry comment
  was almost certainly the Docker proxy returning the same error
  immediately, not the Ollama daemon mid-eviction.

The diagnostic that flipped it: instrumenting `OllamaBackend.generate`
to print env state. `OLLAMA_HOST_env='http://localhost:11434'` showed up
in the failing path but was `None` in the succeeding direct path. Setting
`OLLAMA_HOST=http://localhost:11434` in the otherwise-succeeding path
reproduced the 404; setting `OLLAMA_HOST=http://127.0.0.1:11434` did
not. From there `lsof -nP -iTCP:11434 -sTCP:LISTEN` named Docker as the
hijacker.

## Follow-up — resolved

The Docker port-11434 hijacker investigation was handled in a
follow-up task. The IPv4 pin in `.env` remains the load-bearing fix
either way; this note records the diagnosis, not the root-cause
remediation of the Docker side.
