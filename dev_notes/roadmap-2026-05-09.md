# Roadmap (post-review, 2026-05-09)

Drafted in response to [`analysis-assumptions-2026-05-09.md`](analysis-assumptions-2026-05-09.md).
That review identified eight per-pass concerns and two cross-cutting issues
across the four anomaly passes (`mirror-trade`, `mirror-gap-trends`,
`hs-group-yoy`, `hs-group-trajectory`). This document is the agreed plan
for addressing them, ordered by phase, and the rationale for what we ship
first vs. defer.

The headline editorial commitment running through the plan: **make the
already-implicit caveats first-class data**. Where a pass currently makes
a quietly-stated assumption (a CIF/FOB baseline, a partner being a
transshipment hub, a YoY window crossing a CN8 revision), the new
behaviour records that assumption in the finding's `detail` and emits a
typed caveat. The LLM framing layer (which sits at the end of the plan)
is the consumer: it can lean on these caveats rather than rediscover
them by reading prose.

## Ordering rationale

Three phases, run in order:

1. **Rigour fixes.** Bug-class items, idempotency, and cheap statistical
   corrections. Direct prerequisites for the LLM framing layer reading
   `findings`. Without these, the framing layer would build on
   duplicated rows and silently-wrong values.
2. **Editorial framing.** Reshape the data the LLM (and journalists)
   read so the editorial story is encoded rather than implied.
   Independent of phase 1 in principle, but doing them first means the
   framing layer's prompts can rely on stable input.
3. **LLM framing layer.** The `llm_framing.py` module that turns the
   structured findings into journalist-ready top-lines. Comes last
   because it consumes everything above it.

Phase 4 is a parked-but-named list of statistical depth items we
agreed not to ship now.

## Phase 1 — Rigour fixes

These should land before the LLM framing layer. Most are small.

### 1.1 Idempotent findings with revision history (cross-cut)

**Problem.** Re-running any `--analyse` pass produces duplicate
`findings` rows. The exporters paper over this with `DISTINCT ON`, but
it's brittle and the LLM framing layer reading `findings` directly
would be confused. Worse — even if we deduped, we'd lose the
revision-history story (a YoY value moving when Eurostat revises).

**Approach.** Append + supersede chain, mirroring how `observations`
are versioned.

- Add columns to `findings`:
  - `superseded_at TIMESTAMPTZ` (NULL = current)
  - `superseded_by_finding_id BIGINT REFERENCES findings(id)`
  - `last_confirmed_at TIMESTAMPTZ NOT NULL DEFAULT now()`
- Define a per-subkind natural key (e.g. for `hs_group_yoy`:
  `(subkind, hs_group_ids[1], detail->'windows'->>'current_end')`).
  Codify in a small Python helper, not a generated column — keeps the
  logic visible.
- On insert, look up the existing un-superseded row with the same
  natural key:
  - **Same values**: bump `last_confirmed_at` on the existing row, no
    new insert.
  - **Different values**: insert the new row, set the prior row's
    `superseded_at = now()` and `superseded_by_finding_id` to the new
    id. Same transaction.
- Default queries (Sheets export, briefing pack, future LLM framing)
  filter `WHERE superseded_at IS NULL`.
- A "history" or "diffs" view will eventually surface the chain — out
  of scope here, but the data is there once the chain is recorded.

**Why this and not UNIQUE-with-upsert.** Upsert keeps only the latest;
we want to be able to say "this YoY moved from +12% to +18% on
2026-04-15 — that aligns with Eurostat's correction window for Feb
2026". That's editorially useful. The cost is one extra column and a
bit of insert logic; the value is keeping a class of story we'd
otherwise discard.

**Tests.** Re-run analyser with no underlying data change → no new
rows, `last_confirmed_at` bumped. Re-run with changed data → new row,
prior row marked superseded. Default exporters only see current rows.

### 1.2 Unit-scale parse failure → hard skip (mirror-trade)

**Problem.** When `parse_unit_scale` doesn't recognise a unit format
it falls back to multiplier `1.0` with a WARNING. If the actual scale
was 10⁴ ("10,000 USD") this produces a converted EUR off by 10,000×.
This is a bug, not a limitation.

**Approach.** Treat the fallback as a skip. Log at ERROR with the
unrecognised unit string, mark the row `skipped_unrecognised_unit`,
do not produce a converted value, do not produce a finding from that
release.

**Tests.** Existing parser test covering "10,000 USD" should still
pass. Add a test for an unrecognised format → skip + ERROR log; no
mirror-gap finding emitted.

### 1.3 Theil-Sen slope replaces OLS (hs-group-trajectory)

**Problem.** The accelerating/decelerating distinction uses OLS on a
short smoothed series. A single extreme endpoint can flip the
classification.

**Approach.** Swap OLS for Theil-Sen (median of pairwise slopes).
Robust to endpoint outliers; trivial to implement (numpy or scipy).
Cost is negligible at series lengths of 24-50 windows.

**Tests.** Existing trajectory tests should still pass. Add a test
where injecting a single outlier at the endpoint flips OLS but does
not flip Theil-Sen.

### 1.4 Raise `min_baseline_n` to 6 + emit `low_baseline_n` caveat (mirror-gap-trends)

**Problem.** A z-score from 3 points isn't statistically defensible.
Currently `min_baseline_n=3` means we fire findings on noise.

**Approach.** Default `min_baseline_n=6` (one full window). For
findings firing on fewer than 6 points (because that's all the data
available — early in a partner's coverage), emit but tag with a new
`low_baseline_n` caveat in `caveat_codes`. Don't drop signal silently;
make the noise honest.

**Tests.** Seed exactly 4 baseline points → finding emitted with
`low_baseline_n` caveat. Seed 6+ → finding emitted without it.

### 1.5 kg-coverage metric → conditional decomposition (hs-group-yoy)

**Problem.** Volume-driven vs. price-driven decomposition relies on
`quantity_kg`. For HS groups dominated by pieces (machine tools, EV
cars) or litres (beverages), kg coverage is sparse and the
decomposition produces garbage unit prices.

**Approach.** Compute `kg_coverage_pct = sum(value_eur where
quantity_kg > 0) / sum(value_eur)` over the rolling window. Store in
`detail.totals.kg_coverage_pct`. If below a threshold (default 80%):
- Suppress the volume/price decomposition narrative in the body.
- Set `current_unit_price_eur_per_kg` and `unit_price_pct_change` to
  NULL.
- Add a `low_kg_coverage` caveat.

**Tests.** Seed a group with majority-zero kg → coverage below
threshold → decomposition suppressed, caveat present. Seed a group
with full kg coverage → decomposition produced.

### 1.6 Configurable `--low-base-threshold` (hs-group-yoy)

**Approach.** CLI flag accepting EUR, default unchanged at €50M.
Threshold value recorded in finding `detail.totals.low_base_threshold_eur`
(already there). Trivial.

### 1.7 Trajectory gap detection (hs-group-trajectory)

**Problem.** Trajectory classifier reads `hs_group_yoy` findings.
If the input series has gaps (a YoY pass run with a threshold that
filtered some periods, or one where the underlying Eurostat month
hadn't been ingested), the classifier silently builds a shape from a
discontinuous series.

**Approach.** Before classifying, check the YoY series for period
gaps. If gaps exist:
- Don't classify.
- Emit a `skipped_incomplete_series` finding subkind (or attribute on
  the trajectory finding) with the missing periods listed.
- Surface in Sheets/briefing-pack so the journalist knows.

**Tests.** Seed YoY findings with a 2-month gap → trajectory pass
emits skip finding, no shape classification. Continuous series →
classification proceeds.

## Phase 2 — Editorial framing

These reshape findings to encode editorial context the LLM framing
layer (and journalists reading the briefing pack) need. Independent of
Phase 1 but should land before Phase 3.

### 2.1 Transshipment-hub flag (mirror-trade)

**Approach.** Add a small lookup table `transshipment_hubs` (iso2,
notes, evidence_url, created_by). Seed with NL, BE, HK, SG, AE, MX
based on standard trade-economics evidence. When a `mirror_gap`
finding's partner iso2 is in the table, automatically attach a
`transshipment_hub` caveat and a one-line body annotation
("Netherlands: Rotterdam transshipment effect — gap ≠ direct trade
mismatch").

**Why a table not a list.** Journalists may want to add HK, MO, or
others over time. The evidence_url column gives a citable basis for
each entry rather than baking editorial judgement into code.

### 2.2 CIF/FOB baselines table — lighter version (mirror-trade)

**Approach.** Add `cif_fob_baselines` table with columns: partner_iso2
(NULL = global default), baseline_pct, source, source_url, notes.
Seed with global default 7.5% (current constant). Allow per-partner
overrides as journalists source them. The `mirror-trade` analyser
looks up partner-specific first, then global. Record the
baseline_used_pct in finding `detail` for auditability.

**Out of scope.** Per-(partner, commodity) rows, UNCTAD/WTO mass
backfill. Either of those becomes a research project; flagging here
that we considered and parked them.

**Crucially.** Combined with 2.1, this stops conflating the structural
transshipment effect with the CIF/FOB baseline gap. A 65%+ NL gap is
not "wildly above the 7.5% CIF/FOB baseline" — it's "a transshipment
hub where direct mirror-comparison isn't the right lens".

### 2.3 Multi-partner Eurostat support (cross-cut)

**Problem.** Eurostat queries hardcode `partner = 'CN'`. Goods
reported via Hong Kong / Macau (HK ~15% of China's trade) don't enter
the comparison.

**Approach.** Default behaviour unchanged (`['CN']`). New flag
`--eurostat-partners CN,HK[,MO]` summing the partner rows before
comparison. When >1 partner is used, attach `multi_partner_sum`
caveat. Record the partner list in finding detail.

### 2.4 Configurable trajectory smoothing (hs-group-trajectory)

**Approach.** `--smooth-window N` flag, default 3. Allow N=1 (no
smoothing) for analyses focused on short-term policy effects (tariff
pre-loading spikes etc.). Window value recorded in finding detail.

### 2.5 Seasonality as a feature, not a shape (hs-group-trajectory)

**Approach.** Compute autocorrelation at lag 12 on the raw (unsmoothed)
YoY series. Surface as `detail.features.seasonal_signal_strength`
(a number, e.g. 0.0–1.0). If above a threshold, prepend a body-text
note: "Strong seasonal signal (lag-12 autocorrelation 0.74). YoY shape
should be interpreted as a deseasonalised trend, not raw movement."
Don't add a 13th shape — keeps the shape vocabulary clean for editorial
sorting.

### 2.6 Staleness log line (mirror-gap-trends)

**Approach.** Before running, query the latest `mirror_gap` finding
period. If older than the latest available Eurostat or GACC release
period, log a one-line WARNING: "Latest upstream mirror_gap is
2025-12; Eurostat data extends to 2026-02. Re-run --analyse
mirror-trade first." Don't auto-trigger — auto-triggering hides what's
running.

### 2.7 Single-missing-month tolerance (hs-group-yoy)

**Problem.** Reviewer's proposal 3.1 was full gap-tolerance with
interpolation. I pushed back: interpolating Eurostat is dangerous
given the publication-lag pattern. Compromise:

**Approach.** Allow up to 1 missing month in either window (most-recent
month often lags). Sum what's there; do not interpolate. Record
`detail.totals.n_months_used` and `detail.totals.missing_months`. If
1 month missing, emit `partial_window` caveat. If 2+ missing, skip the
window entirely (current behaviour).

### 2.8 Blanket `cn8_revision` caveat for cross-year-boundary windows

**Problem.** Reviewer's proposal 3.4 — full CN8 concordance table —
is a research project. Interim cheap-honesty step:

**Approach.** Apply a `cn8_revision` caveat automatically to any YoY
window spanning a calendar-year boundary (which is most of them, since
Eurostat revises CN8 annually each January). Add the caveat row with
a clear summary explaining the limitation. Flag full concordance
table as parked work in Phase 4.

## Phase 3 — LLM framing layer

The original "next step" before this review surfaced. Now scoped to
operate on the cleaner Phase 1+2 finding set.

### 3.1 `llm_framing.py` v1

**Approach.** Module reads `findings WHERE superseded_at IS NULL`,
clusters by group / partner / story-shape, asks a local Ollama model
to draft a short journalist-ready top-line per cluster. Outputs new
findings with `kind='llm_topline', subkind='narrative'` so they sit
alongside the deterministic findings rather than over-writing them.

**Discipline.** Every numeric claim in the LLM output must be
verifiable against an underlying finding's `detail` block. If the LLM
emits a number that doesn't match, the framing pipeline rejects it
and re-prompts. The LLM never computes — it narrates.

**Inputs the layer relies on (from Phase 1+2).**
- Caveat codes attached to each finding so it can hedge correctly.
- `superseded_at` so it doesn't draft narratives over revised data.
- kg-coverage suppression so it doesn't claim "volume-driven" when it
  isn't.
- Transshipment / CIF-FOB framing so it doesn't say "Italy's mirror gap
  matches Netherlands'" when one is structural and one isn't.

### 3.2 Briefing-pack integration

**Approach.** New "Top-line narratives" section above the
deterministic sections in the briefing pack, sourced from
`llm_topline` findings. The deterministic sections remain — the
narrative section is editorial framing, not replacement.

## Phase 4 — Parked statistical depth

Named here so we don't lose them; not on the immediate path.

- **Per-route CIF/FOB research.** UNCTAD/WTO sourcing for partner ×
  commodity-group baselines. Pick up if a story specifically rests on
  a CIF/FOB-sensitive comparison.
- **Structural-break detection (Chow / CUSUM)** in mirror-gap-trends
  baselines. Statistically right but unstable on the 11 periods we
  have. Revisit when we have ~24 periods.
- **CN8 concordance table.** Full mapping of old→new codes across
  Eurostat's annual revisions. Pick up when a journalist hits a story
  where the cross-year comparison bites.

## Suggested commit cadence

- One commit per Phase 1 item (7 commits): each is small enough to
  review in isolation.
- Phase 2 items can pair where they share a table or migration (e.g.
  2.1 + 2.2 land together since they both add lookup tables and both
  reshape mirror-gap findings).
- Phase 3 lands as 2-3 commits: the framing module, then briefing-pack
  integration, then any prompt iteration.
- Schema changes get their own commits with the SQL clearly visible
  (we're still pre-Alembic; one CREATE COLUMN per commit is the rule).
