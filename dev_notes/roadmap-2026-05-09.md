# Roadmap (post-review, 2026-05-09)

Drafted in response to [`analysis-assumptions-2026-05-09.md`](analysis-assumptions-2026-05-09.md).
That review identified eight per-pass concerns and two cross-cutting issues
across the four anomaly passes (`mirror-trade`, `mirror-gap-trends`,
`hs-group-yoy`, `hs-group-trajectory`). This document is the agreed plan
for addressing them, ordered by phase, and the rationale for what we ship
first vs. defer.

The headline editorial commitment running through the plan: **make the
already-implicit caveats first-class data**. Where a pass currently makes
a quietly-stated assumption (a CIF/FOB baseline, a partner being a
transshipment hub, a YoY window crossing a CN8 revision), the new
behaviour records that assumption in the finding's `detail` and emits a
typed caveat. The LLM framing layer (which sits at the end of the plan)
is the consumer: it can lean on these caveats rather than rediscover
them by reading prose.

## Ordering rationale

Three phases, run in order:

1. **Rigour fixes.** Bug-class items, idempotency, and cheap statistical
   corrections. Direct prerequisites for the LLM framing layer reading
   `findings`. Without these, the framing layer would build on
   duplicated rows and silently-wrong values.
2. **Editorial framing.** Reshape the data the LLM (and journalists)
   read so the editorial story is encoded rather than implied.
   Independent of phase 1 in principle, but doing them first means the
   framing layer's prompts can rely on stable input.
3. **LLM framing layer.** The `llm_framing.py` module that turns the
   structured findings into journalist-ready top-lines. Comes last
   because it consumes everything above it.

Phase 4 is a parked-but-named list of statistical depth items we
agreed not to ship now.

### 1.1 Idempotent findings with revision history (cross-cut)

**Shipped: `8f18e68`.** Five new columns on findings
(`natural_key_hash`, `value_signature`, `superseded_at`,
`superseded_by_finding_id`, `last_confirmed_at`) plus a partial unique
index `(natural_key_hash) WHERE superseded_at IS NULL` enforcing
DB-level "at most one active finding per natural key". New
`findings_io.py` module: `emit_finding()` returns `(id, action ∈
{inserted_new, confirmed_existing, superseded})`; per-subkind natural
keys (`nk_mirror_gap`, `nk_hs_group_yoy`, etc.) centralised there. All
four analyser call sites migrated; backfill script stamped 1245
existing findings.

**Problem.** Re-running any `--analyse` pass produces duplicate
`findings` rows. The exporters paper over this with `DISTINCT ON`, but
it's brittle and the LLM framing layer reading `findings` directly
would be confused. Worse — even if we deduped, we'd lose the
revision-history story (a YoY value moving when Eurostat revises).

**Approach.** Append + supersede chain, mirroring how `observations`
are versioned.

- Add columns to `findings`:
  - `superseded_at TIMESTAMPTZ` (NULL = current)
  - `superseded_by_finding_id BIGINT REFERENCES findings(id)`
  - `last_confirmed_at TIMESTAMPTZ NOT NULL DEFAULT now()`
- Define a per-subkind natural key (e.g. for `hs_group_yoy`:
  `(subkind, hs_group_ids[1], detail->'windows'->>'current_end')`).
  Codify in a small Python helper, not a generated column — keeps the
  logic visible.
- On insert, look up the existing un-superseded row with the same
  natural key:
  - **Same values**: bump `last_confirmed_at` on the existing row, no
    new insert.
  - **Different values**: insert the new row, set the prior row's
    `superseded_at = now()` and `superseded_by_finding_id` to the new
    id. Same transaction.
- Default queries (Sheets export, briefing pack, future LLM framing)
  filter `WHERE superseded_at IS NULL`.
- A "history" or "diffs" view will eventually surface the chain — out
  of scope here, but the data is there once the chain is recorded.

**Why this and not UNIQUE-with-upsert.** Upsert keeps only the latest;
we want to be able to say "this YoY moved from +12% to +18% on
2026-04-15 — that aligns with Eurostat's correction window for Feb
2026". That's editorially useful. The cost is one extra column and a
bit of insert logic; the value is keeping a class of story we'd
otherwise discard.

**Tests.** Re-run analyser with no underlying data change → no new
rows, `last_confirmed_at` bumped. Re-run with changed data → new row,
prior row marked superseded. Default exporters only see current rows.

### 1.2 Unit-scale parse failure → hard skip (mirror-trade)

**Shipped: `69086c9`.** `parse_unit_scale` now distinguishes
"missing unit" `(1.0, None)` from "unrecognised non-empty unit"
`(None, None)`. Caller skips on `None` with a new
`skipped_unrecognised_unit` tally key. Logs at ERROR. Live data has
zero unrecognised units; the protection is for future extensions.

**Problem.** When `parse_unit_scale` doesn't recognise a unit format
it falls back to multiplier `1.0` with a WARNING. If the actual scale
was 10⁴ ("10,000 USD") this produces a converted EUR off by 10,000×.
This is a bug, not a limitation.

**Approach.** Treat the fallback as a skip. Log at ERROR with the
unrecognised unit string, mark the row `skipped_unrecognised_unit`,
do not produce a converted value, do not produce a finding from that
release.

**Tests.** Existing parser test covering "10,000 USD" should still
pass. Add a test for an unrecognised format → skip + ERROR log; no
mirror-gap finding emitted.

### 1.3 Theil-Sen slope replaces OLS (hs-group-trajectory)

**Shipped: `0bc9cb9`.** `_linear_slope` replaced with
`_theil_sen_slope` (median of pairwise slopes). Method bumped to
`hs_group_trajectory_v3_theil_sen_slope`. Live impact: Solar PV
cells & modules trajectory flipped from `falling` → `falling_decelerating`
(real editorial revision — the decline is slowing, not just continuing).

**Problem.** The accelerating/decelerating distinction uses OLS on a
short smoothed series. A single extreme endpoint can flip the
classification.

**Approach.** Swap OLS for Theil-Sen (median of pairwise slopes).
Robust to endpoint outliers; trivial to implement (numpy or scipy).
Cost is negligible at series lengths of 24-50 windows.

**Tests.** Existing trajectory tests should still pass. Add a test
where injecting a single outlier at the endpoint flips OLS but does
not flip Theil-Sen.

### 1.4 Raise `min_baseline_n` to 6 + emit `low_baseline_n` caveat (mirror-gap-trends)

**Shipped: `05f896c`.** Implemented as **two thresholds**: hard floor
`min_baseline_n` stays 3 (mathematical minimum), confidence threshold
`LOW_BASELINE_N_THRESHOLD = 6` triggers the `low_baseline_n` caveat.
"Make the noise honest" rather than dropping early-period signal.
Same commit also added `method` to all four analysers' value_signature
dicts so version bumps propagate as supersedes — necessary plumbing
for any future Phase 1.X edit that doesn't move numbers but does
change editorial substance. Live: 4 of 11 z-score findings (n=5
baselines for BLOC:eu_bloc, DE, FR, IT in 2025-08) now carry the
caveat.

**Problem.** A z-score from 3 points isn't statistically defensible.
Currently `min_baseline_n=3` means we fire findings on noise.

**Approach.** Default `min_baseline_n=6` (one full window). For
findings firing on fewer than 6 points (because that's all the data
available — early in a partner's coverage), emit but tag with a new
`low_baseline_n` caveat in `caveat_codes`. Don't drop signal silently;
make the noise honest.

**Tests.** Seed exactly 4 baseline points → finding emitted with
`low_baseline_n` caveat. Seed 6+ → finding emitted without it.

### 1.5 kg-coverage metric → conditional decomposition (hs-group-yoy)

**Shipped: `0c4656c`.** `_hs_group_per_period_totals` extended to
return `eur_with_kg` per period; analyser sums it over the rolling
window to compute `kg_coverage_pct`. Threshold:
`KG_COVERAGE_DECOMPOSITION_THRESHOLD = 0.80`. Below it: unit-price
fields nulled, `low_kg_coverage` caveat attached, body annotation
explains the suppression. Method bumped to
`hs_group_yoy_v5_kg_coverage_gating`. Empirical note: every HS group
in the live DB sits at 99-100% kg coverage on real Eurostat data —
the protection is defensive against a failure mode that doesn't
currently fire, but the test proves the gating works and the live
confirms decomposition narratives are well-grounded.

**Problem.** Volume-driven vs. price-driven decomposition relies on
`quantity_kg`. For HS groups dominated by pieces (machine tools, EV
cars) or litres (beverages), kg coverage is sparse and the
decomposition produces garbage unit prices.

**Approach.** Compute `kg_coverage_pct = sum(value_eur where
quantity_kg > 0) / sum(value_eur)` over the rolling window. Store in
`detail.totals.kg_coverage_pct`. If below a threshold (default 80%):
- Suppress the volume/price decomposition narrative in the body.
- Set `current_unit_price_eur_per_kg` and `unit_price_pct_change` to
  NULL.
- Add a `low_kg_coverage` caveat.

**Tests.** Seed a group with majority-zero kg → coverage below
threshold → decomposition suppressed, caveat present. Seed a group
with full kg coverage → decomposition produced.

### 1.6 Configurable `--low-base-threshold` (hs-group-yoy)

**Shipped: `5de2270`.** CLI flag flows through to both
`detect_hs_group_yoy` and `detect_hs_group_trajectories`. Threshold
value already round-tripped via `detail.totals.low_base_threshold_eur`
(yoy) and `detail.features.low_base_threshold_eur` (trajectory).
Re-runs that flip any group's low-base classification trigger
supersedes via the existing value_signature mechanism.

**Approach.** CLI flag accepting EUR, default unchanged at €50M.
Threshold value recorded in finding `detail.totals.low_base_threshold_eur`
(already there). Trivial.

### 1.7 Trajectory gap detection (hs-group-trajectory)

**Shipped: `3dc4c72`.** New `_detect_series_gaps()` helper checks
the YoY input series for missing months before classification. If
gaps exist, the group is tallied under `skipped_incomplete_series`
and no trajectory finding is written. Same commit also added
`AND superseded_at IS NULL` to `_fetch_group_yoy_series` so
trajectory always builds on the current revision (Phase 1.1 hygiene).
Live data is gap-free; the protection guards against future ingestion
gaps or threshold-filtered yoy passes.

**Problem.** Trajectory classifier reads `hs_group_yoy` findings.
If the input series has gaps (a YoY pass run with a threshold that
filtered some periods, or one where the underlying Eurostat month
hadn't been ingested), the classifier silently builds a shape from a
discontinuous series.

**Approach.** Before classifying, check the YoY series for period
gaps. If gaps exist:
- Don't classify.
- Emit a `skipped_incomplete_series` finding subkind (or attribute on
  the trajectory finding) with the missing periods listed.
- Surface in Sheets/briefing-pack so the journalist knows.

**Tests.** Seed YoY findings with a 2-month gap → trajectory pass
emits skip finding, no shape classification. Continuous series →
classification proceeds.

### 2.1 Transshipment-hub flag (mirror-trade)

**Shipped: `c0aa48c`** (paired with 2.2). New `transshipment_hubs`
table seeded with NL, BE, HK, SG, AE, MX, each with a citable
`evidence_url`. mirror-trade analyser auto-attaches a
`transshipment_hub` caveat and body annotation when the partner is
in the table; aggregates skip the lookup. Live: 11 NL findings now
carry the caveat, editorially separating Rotterdam transshipment
from CIF/FOB baseline.

**Approach.** Add a small lookup table `transshipment_hubs` (iso2,
notes, evidence_url, created_by). Seed with NL, BE, HK, SG, AE, MX
based on standard trade-economics evidence. When a `mirror_gap`
finding's partner iso2 is in the table, automatically attach a
`transshipment_hub` caveat and a one-line body annotation

mismatch").

**Why a table not a list.** Journalists may want to add HK, MO, or
others over time. The evidence_url column gives a citable basis for
each entry rather than baking editorial judgement into code.

### 2.2 CIF/FOB baselines table — lighter version (mirror-trade)

**Shipped: `c0aa48c`** (paired with 2.1). New `cif_fob_baselines`
table seeded with the global default (7.5%, UNCTAD provenance).
mirror-trade looks up per-partner first, falls back to global. Each
finding records `detail.cif_fob_baseline = {baseline_pct, scope,
partner_iso2, source, source_url, baseline_id}` so the editorial
basis is auditable. Method bumped to
`mirror_trade_v2_lookup_baselines_and_hubs`. Per-route research
deferred to Phase 4 as planned.

**Approach.** Add `cif_fob_baselines` table with columns: partner_iso2
(NULL = global default), baseline_pct, source, source_url, notes.
Seed with global default 7.5% (current constant). Allow per-partner
overrides as journalists source them. The `mirror-trade` analyser
looks up partner-specific first, then global. Record the
baseline_used_pct in finding `detail` for auditability.

**Out of scope.** Per-(partner, commodity) rows, UNCTAD/WTO mass
backfill. Either of those becomes a research project; flagging here
that we considered and parked them.

**Crucially.** Combined with 2.1, this stops conflating the structural
transshipment effect with the CIF/FOB baseline gap. A 65%+ NL gap is
not "wildly above the 7.5% CIF/FOB baseline" — it's "a transshipment
hub where direct mirror-comparison isn't the right lens".

### 2.3 Multi-partner Eurostat support (cross-cut)

**Shipped: `b14f468`.** Default unchanged (`['CN']`); CLI flag
`--eurostat-partners CN,HK[,MO]` sums across the listed partners.
When >1 is used, findings carry a `multi_partner_sum` caveat and a
body annotation about methodological non-comparability.
`detail.eurostat.partners_summed` round-trips the exact list. Method
bumped to `mirror_trade_v3_multi_partner`. Operational gap noted:
the live DB has zero HK/MO partner rows (ingestion has only ever
pulled `partner=CN`); re-ingest with `--partner CN --partner HK
--partner MO` is a separate operational task per investigation.

**Problem.** Eurostat queries hardcode `partner = 'CN'`. Goods
reported via Hong Kong / Macau (HK ~15% of China's trade) don't enter
the comparison.

**Approach.** Default behaviour unchanged (`['CN']`). New flag
`--eurostat-partners CN,HK[,MO]` summing the partner rows before
comparison. When >1 partner is used, attach `multi_partner_sum`
caveat. Record the partner list in finding detail.

### 2.4 Configurable trajectory smoothing (hs-group-trajectory)

**Shipped: `6aa9ad6`** (paired with 2.5). `--smooth-window N` CLI
flag, default 3 (historic behaviour). `features.smoothing_window`
records the value used per finding. Pass `--smooth-window 1` to
disable smoothing entirely for analyses focused on short-term policy

**Approach.** `--smooth-window N` flag, default 3. Allow N=1 (no
smoothing) for analyses focused on short-term policy effects (tariff

### 2.5 Seasonality as a feature, not a shape (hs-group-trajectory)

**Shipped: `6aa9ad6`** (paired with 2.4). New
`_autocorrelation_at_lag()` helper computes a *detrended* Pearson
correlation between `ys[:-lag]` and `ys[lag:]` — Theil-Sen detrend
prevents monotonic series triggering false-positive lag-12
autocorrelation. Surfaces as `features.seasonal_signal_strength`,
`has_strong_seasonal_signal` (POSITIVE-only gate, since negative
lag-12 correlation is a U/inverse-U detrend artefact, not
seasonality). Body adds a 📅 annotation when the flag fires; shape
vocabulary stays at 12 entries. Method bumped to
`hs_group_trajectory_v4_seasonal_feature_and_configurable_smoothing`.
Live: 0/30 active trajectories carry the flag (post-COVID shapes are
pandemic-cycle, not calendar-seasonal).

**Approach.** Compute autocorrelation at lag 12 on the raw (unsmoothed)
YoY series. Surface as `detail.features.seasonal_signal_strength`
(a number, e.g. 0.0–1.0). If above a threshold, prepend a body-text
note: "Strong seasonal signal (lag-12 autocorrelation 0.74). YoY shape
should be interpreted as a deseasonalised trend, not raw movement."
Don't add a 13th shape — keeps the shape vocabulary clean for editorial
sorting.

### 2.6 Staleness log line (mirror-gap-trends)

**Shipped: `a7a19ce`.** New `_log_mirror_gap_staleness()` runs
before `detect_mirror_gap_trends()`. Three branches: INFO "OK" when
fully fresh, INFO "no findings yet" when upstream hasn't run, WARNING
with exact periods + action ("Re-run --analyse mirror-trade first")
when stale. Live: WARNING currently fires because GACC has 2026-03
data but mirror_gap findings stop at 2026-02 — exactly the kind of
state the journalist should see surfaced.

**Approach.** Before running, query the latest `mirror_gap` finding
period. If older than the latest available Eurostat or GACC release
period, log a one-line WARNING: "Latest upstream mirror_gap is
2025-12; Eurostat data extends to 2026-02. Re-run --analyse
mirror-trade first." Don't auto-trigger — auto-triggering hides what's
running.

### 2.7 Single-missing-month tolerance (hs-group-yoy)

**Shipped: `26b2c94`** (paired with 2.8). Allow up to 1 missing
month across both 12mo windows combined; sum what's present; never
interpolate. `detail.totals` records `partial_window`,
`missing_months_current`, `missing_months_prior`, `n_months_used_*`.
Caveat `partial_window` attached when triggered. 2+ missing months
still skip. Live: +17 import + +15 export new findings on the
leading-edge anchor that previously got dropped, all
`partial_window=true`.

**Problem.** Reviewer's proposal 3.1 was full gap-tolerance with
interpolation. I pushed back: interpolating Eurostat is dangerous
given the publication-lag pattern. Compromise:

**Approach.** Allow up to 1 missing month in either window (most-recent
month often lags). Sum what's there; do not interpolate. Record
`detail.totals.n_months_used` and `detail.totals.missing_months`. If
1 month missing, emit `partial_window` caveat. If 2+ missing, skip the
window entirely (current behaviour).

### 2.8 Blanket `cn8_revision` caveat for cross-year-boundary windows

**Shipped: `26b2c94`** (paired with 2.7). `cn8_revision` caveat
auto-applied when `start_prior.year != end_curr.year`. Method bumped
to `hs_group_yoy_v6_partial_window_and_cn8_revision_caveats`. Live:
all 1181 active hs_group_yoy* findings carry the caveat (every
24-month window in the data spans a year boundary). Full concordance

**Problem.** Reviewer's proposal 3.4 — full CN8 concordance table —

**Approach.** Apply a `cn8_revision` caveat automatically to any YoY

### 3.1 `llm_framing.py` v1

**Shipped: `e3766c7`.** v1 generates 2-3 sentence editorial top-lines
per HS group, drawing on the latest active hs_group_yoy and
hs_group_trajectory findings (both flows). Subkind
`narrative_hs_group`, natural key `(hs_group_id,)`, kind
`llm_topline`, method `llm_topline_v1_hs_group`. Default backend
Ollama, default model `qwen3.6:latest` — passes `think=False` so
thinking models skip the reasoning channel and write directly to
content (qwen3.x emits ~10k chars of reasoning per prompt otherwise,
exhausting any reasonable num_predict cap). FakeBackend in tests
keeps Ollama out of CI.

**Numeric verification discipline.** Every number extracted from the
LLM output must match a fact within tolerance: percentages ±0.5pp
(sign-aware first, magnitude-only fallback for cross-clause prose
ambiguity), currency ±5% relative, counts exact. Calendar years
(19xx/20xx), time periods ("12 months", "24-month") and HS codes
pre-stripped from extraction — editorial scaffolding, not facts.
Word-boundary lookbehind on the regex prevents "CN8" parsing as
"8". Currency values not sign-flipped by decrease verbs (stocks vs
flows). On verification failure: REJECT the narrative, log a
WARNING, increment `skipped_unverified`. Editorial cost: silence on
that group. Editorial benefit: never confidently wrong.

**Live result.** 15/16 hs_groups produce verified narratives on
first pass (the 16th has no underlying findings). One hallucination
correctly rejected on the first run: qwen3.6 cited "93%" for
permanent magnets — recalled from training data (Lisa O'Carroll's
piece), NOT in our facts; the verifier caught it; the finding was
silently dropped.

**Approach.** Module reads `findings WHERE superseded_at IS NULL`,
clusters by group / partner / story-shape, asks a local Ollama model
to draft a short journalist-ready top-line per cluster. Outputs new
findings with `kind='llm_topline', subkind='narrative'` so they sit
alongside the deterministic findings rather than over-writing them.

**Discipline.** Every numeric claim in the LLM output must be
verifiable against an underlying finding's `detail` block. If the LLM
emits a number that doesn't match, the framing pipeline rejects it
and re-prompts. The LLM never computes — it narrates.

**Inputs the layer relies on (from Phase 1+2).**
- Caveat codes attached to each finding so it can hedge correctly.
- `superseded_at` so it doesn't draft narratives over revised data.
- kg-coverage suppression so it doesn't claim "volume-driven" when it
  isn't.
- Transshipment / CIF-FOB framing so it doesn't say "Italy's mirror gap
  matches Netherlands'" when one is structural and one isn't.

## Phase 5 — Methodology audit + clean-state rebuild ✓ DONE

Triggered 2026-05-09 by a strategic review of whether the tool was
genuinely surfacing newsworthy insights vs converging on a
self-consistent loop with confirmation bias. The review surfaced six
concerns; this phase addressed three of them (HK/MO routing,
historical baseline depth, methodology validation against known
shocks) and laid groundwork for the others (sector breadth,
threshold robustness, lead-vs-narrative framing).

### 5.1 HK/MO partner inclusion (cross-cut)

**Problem.** Eurostat reports goods routed via Hong Kong / Macau
under partner=HK / partner=MO rather than partner=CN. Phase 2.3 had
added analyser-side support for `--eurostat-partners CN,HK[,MO]` but
no HK/MO data had ever been ingested. So mirror-trade was
structurally biased and the broader hs-group analyses entirely missed
HK-routed Chinese trade (~15% of total Chinese exports to EU).

**Approach.** Two changes layered:

1. **Multi-partner default everywhere.** New constant
   `EUROSTAT_PARTNERS_DEFAULT = ('CN', 'HK', 'MO')` in `anomalies.py`,
   adopted as the default by all four analysers (mirror-trade,
   hs-group-yoy, hs-group-trajectory, llm-framing). CLI override
   `--eurostat-partners CN` available for the narrower direct-China
   view (matches Soapbox/Merics single-partner figures). The
   `multi_partner_sum` caveat fires by default as honest annotation.

2. **Plumbed through helpers.** Three SQL helpers in `anomalies.py`
   (`_hs_group_per_period_totals`, `_hs_group_top_cn8s`,
   `_hs_group_top_reporters`) had `AND partner = 'CN'` hardcoded;
   replaced with `AND partner = ANY(%s)` taking the partners tuple.

Method versions bumped: `mirror_trade_v3` → `v4_multi_partner_default`;
`hs_group_yoy_v6` → `v7_multi_partner_default`;
`hs_group_trajectory_v4` → `v5_inherits_multi_partner_yoy`. The
existing supersede mechanism propagates the bumps as natural-key
supersedes when the analysers re-run.

### 5.2 upsert_observations partner-scoped fast path

**Problem.** The bulk-insert fast path in `db.upsert_observations`
keyed off "release_id has any rows" — adding HK/MO observations to
existing CN-only releases fell into the per-row slow path, taking
~6 minutes per period instead of ~1 second.

**Approach.** Scope the freshness check by `partner_country`. If
none of the incoming observations' partners already have rows under
this release, take the fast path; otherwise fall back to the slow
path. Falls back to release-level check when any incoming
observation has NULL partner (e.g. some GACC totals).
~280x speedup on partner-additive ingest. Covered by new test
`test_partner_additive_takes_fast_path`.

### 5.3 GACC parser — historical title formats

**Problem.** GACC release titles in 2018 and earlier omit the
leading `(N)` section prefix (e.g. "China's Total Export & Import
Values by Country/Region, July 2018 (in CNY)"); use "RMB" as a
synonym for CNY; and have a "(Only August, in CNY)" parenthetical
variant for some monthly releases. The original regex was tested
only against 2025/2026 formats and failed every 2018 release.

**Approach.** Optional `(N)` prefix in the regex; section number
inferred from the description when absent ("by Country/Region" → 4,
"by Trade Mode" → 2, default → 1); "RMB" accepted and normalised to
"CNY"; "Only X" parenthetical absorbed in the currency capture
group. Covered by new test `test_metadata_handles_2018_historical_formats`.

### 5.4 Historical Eurostat + GACC backfill

**Eurostat range.** Previously 2021-01 → 2026-02 (62 periods,
~7.5M raw rows, partner=CN only). Re-ingested 2017-01 → 2026-02
(110 periods, partners CN+HK+MO). Bulk file pattern
`full_v2_YYYYMM.7z` works back to 2017; older months not probed.

**GACC range.** Previously 12 months (Mar 2025 → Mar 2026, partner
preliminary index only). Now walks 9 yearly indexes
(`preliminary.html` for current year + `preliminaryYYYY.html` for
2018–2025). Section 4 (by Country/Region) parses cleanly across all
years after the parser fix above; sections 1, 2, 3 still raise
`NotImplementedError` and are skipped (not section 4 = not in scope).

### 5.5 Clean-state rebuild

Wiped the live DB, re-applied `schema.sql` from scratch, archived
the `migrations/` directory (folded into `schema.sql`; preserved as
`migrations.archived-2026-05-09/` for the dev history), and
re-ingested everything in one pass with the new defaults. Rationale:
the supersede chains in the live DB were almost entirely development
churn (method version bumps from our own iterations), not editorially
useful revision history. With no journalists yet using the tool,
"start fresh from the canonical schema" was cleaner than continuing
to layer migrations.

### 5.6 Pre-registered shock validation

**Document.** `dev_notes/shock-validation-2026-05-09.md` —
pre-registered expectations for what the analysers should surface
across four known historical shocks (2018 Section 232 steel/aluminium
tariffs, Q1 2020 COVID lockdown, Feb 2022 Russia invasion → renewables
substitution, Oct 2023 EU EV anti-subsidy probe). Written **before**
running the analysers; Results sections filled in afterwards. The

**Validation queries.** `dev_notes/shock-validation-queries.sql` —

### 5.7 Things this phase deliberately did NOT address

The strategic review surfaced six concerns; Phase 5 fully addressed
three (HK/MO, historical depth, validation methodology) and partially

### 6.0.5 EU-27 means EU-27 at all times ✓ DONE

### 6.0.6 ~10000x analyser speedup ✓ DONE

**Shipped: `70d7bc5`.** A re-run of all hs-group-yoy findings
(needed to get the v8 method tag onto the live findings) initially
took ~40 minutes for ~40% completion — projected to take ~3 hours
because each per-anchor query was doing a Parallel Seq Scan on
the 17.5M-row `eurostat_raw_rows` table.

Two changes together drop the planner's estimated cost from
~476725 to ~41:

1. New covering index `idx_eu_raw_analyser` on
   `(flow, partner, product_nc text_pattern_ops, period)` INCLUDE
   `(value_eur, quantity_kg, reporter)`. Added to `schema.sql`.
2. New helper `_hs_pattern_or_clause(patterns)` rewrites
   `product_nc LIKE ANY(%s)` (which the planner refuses to push down
   through the text_pattern_ops btree) into separate ORed LIKEs
   that the planner happily turns into a BitmapOr.

Wall time on the re-run dropped from ~40 minutes (mid-run, ~40%)
to ~7 minutes (full chain incl. flow=2, trajectory, llm-framing,
briefing pack regeneration).

### 6.0.7 trajectory tolerates gaps ✓ DONE

**Shipped: `13f5ea1`.** Phase 1.7's all-or-nothing gap rejection
was producing only ~5/58 expected trajectory findings on real data
(~9% coverage) — including missing the EV-probe `inverse_u_peak`
shape that the shock validation had pre-registered as the cleanest
test case.

Replace with longest-contiguous-run: find the longest unbroken
sub-series and classify on that. Honest because the chosen window
is recorded in `features.{effective_first_period,
effective_last_period, original_series_length,
effective_series_length, dropped_periods_due_to_gaps}`. The
`TRAJECTORY_MIN_WINDOWS = 6` safeguard still rejects too-short
remnants under `skipped_insufficient_data`.

Coverage went from **5 to 57 trajectory findings** (29 + 28 across
flow=1 and flow=2). The EV trajectory now classifies as
`dip_recovery` with trough at 2024-08, exactly when EU duties bit
hardest. Method bump: `hs_group_trajectory v6→v7_longest_contiguous_run`.
